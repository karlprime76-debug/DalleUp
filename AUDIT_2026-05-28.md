## 1. Vue d'ensemble

DalleUp est une application de livraison de repas multi-rôles :

- **Client** : inscription, panier, checkout, commandes, adresses, profil, suppression de compte.
- **Restaurant** : onboarding, dashboard, menu, catégories, commandes, profil, paramètres, facturation, wallet.
- **Livreur** : dashboard, profil, disponibilité, livraisons, wallet, gains, statuts de validation.
- **Admin** : validations, restaurants, livreurs, paiements, payouts, notifications, audit, paramètres, sponsoring, conformité.

L'application a significativement progressé depuis le précédent audit : CSP ajoutée, pagination partielle, suppression de compte, synchronisation partielle KYC en base, rate limit appliqué sur plusieurs endpoints sensibles, signed URLs réduites à 15 minutes.

## 2. Sécurité

### 2.1 Points positifs

- **Authentification** : NextAuth v4 avec credentials, JWT, bcryptjs, Prisma adapter.
- **Autorisation pages** : `middleware.ts` protège `/app`, `/restaurant`, `/driver`, `/admin` par rôle.
- **Autorisation API** : guards centralisés (`requireAdminApi`, `requireRestaurantApi`, `requireApprovedDriverApi`).
- **Rate limiting présent** sur inscription, login credentials, reset password, forgot password, suppression de compte et admin API.
- **Headers de sécurité** dans `next.config.ts` : `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `HSTS`, `Content-Security-Policy`.
- **Reset password** : token crypto 32 bytes, expiration 30 min, suppression des tokens expirés, réponse neutre si email inconnu.
- **Logs d'inscription améliorés** : plus d'email/rôle en clair dans les logs d'erreur.
- **Suppression de compte** présente via `DELETE /api/account`.
- **PayDunya IPN** : vérification de hash avec `timingSafeEqual`.

### 2.2 Risques et recommandations

| Sévérité | Description | Localisation | Recommandation |
|----------|-------------|--------------|----------------|
| **Élevée** | Rate limiting encore en mémoire (`Map`) : non partagé entre instances, perdu au redémarrage, contournable en serverless/multi-instance | `src/lib/rate-limit.ts` | Migrer vers Redis/Upstash ou stockage persistant |
| **Élevée** | `setInterval(cleanup)` au niveau module peut garder le process actif ou poser problème en environnement serverless | `src/lib/rate-limit.ts:30` | Éviter `setInterval`, faire un cleanup opportuniste par requête ou externaliser |
| **Élevée** | Suppression de compte supprime physiquement commandes, paiements, audit et restaurants : risque légal/comptable et casse l'historique | `src/app/api/account/route.ts` | Préférer anonymisation + conservation comptable/audit |
| **Moyenne** | CSP contient encore `'unsafe-inline'` et `'unsafe-eval'` pour les scripts | `next.config.ts` | Durcir par environnement, nonce/hash pour scripts inline |
| **Moyenne** | `NEXTAUTH_SECRET` n'est pas validé explicitement au démarrage | Config env | Ajouter une validation stricte des variables critiques |
| **Moyenne** | Endpoint PayDunya callback accepte uniquement `formData(data)` ; si PayDunya envoie JSON direct, l'IPN échoue | `src/app/api/payments/paydunya/callback/route.ts` | Supporter form-data et JSON selon la spec PayDunya |
| **Moyenne** | `APP_URL` PayDunya fallback vers production | `src/lib/payments/paydunya.ts:49` | Supprimer le fallback prod, exiger `APP_URL` |
| **Faible** | Pas de rate limit spécifique sur checkout PayDunya, commandes et uploads | API publiques/sensibles | Ajouter des limites par route métier |

## 3. Architecture & base de données

### 3.1 Points positifs

- **Schéma Prisma riche** : utilisateurs, restaurants, menu, commandes, livraisons, paiements, wallets, ledger, notifications, sponsoring, paramètres plateforme.
- **Contraintes améliorées** : `Address` possède maintenant `@@unique([userId, street, city])`.
- **DriverStatus clarifié** : `driverStatus` est non-nullable avec `@default(PENDING)`.
- **KYC partiellement indexé en DB** : `verificationStatus` et `verificationSubmittedAt` ajoutés sur `User`.
- **Audit admin** : modèle `AdminAuditLog` avec index.
- **Wallet/Ledger** : logique de crédit restaurant/livreur séparée et idempotence partielle via `PlatformRevenue`.

### 3.2 Risques et recommandations

| Sévérité | Description | Localisation | Recommandation |
|----------|-------------|--------------|----------------|
| **Élevée** | `Payout` reste isolé du ledger : aucune relation vers `Wallet`, `LedgerEntry` ou `SettlementBatch` | `prisma/schema.prisma:591-601` | Lier chaque payout à un wallet et à des écritures ledger |
| **Élevée** | `SettlementBatch` n'a aucune relation avec les entrées réglées | `prisma/schema.prisma:638-647` | Ajouter `settlementBatchId` sur `LedgerEntry` ou table de jointure |
| **Moyenne** | `LedgerEntry.orderId` n'a pas de relation Prisma vers `Order` | `prisma/schema.prisma:564-575` | Ajouter relation optionnelle pour intégrité référentielle |
| **Moyenne** | Plusieurs relations importantes n'ont pas `onDelete` explicite, ce qui rend la suppression manuelle fragile | `Restaurant`, `Order`, `Delivery`, `Payment`, etc. | Définir une stratégie claire : restrict, cascade ou anonymisation |
| **Moyenne** | KYC reste principalement stocké en manifests JSON Supabase | `src/lib/supabase/verification-storage.ts` | Créer une table `VerificationRequest` / `VerificationDocument` |
| **Faible** | `BillingNotification` accepte `userId` et `restaurantId` tous deux null | `prisma/schema.prisma:373-389` | Ajouter validation métier stricte ou contrainte applicative |

## 4. API routes & logique métier

### 4.1 Points positifs

- **Commandes client paginées** : `GET /api/orders` supporte `page`/`limit`.
- **Payouts admin paginés** : `GET /api/admin/payouts`.
- **Commandes** : numéro plus robuste (`Date.now().toString(36)` + random bytes).
- **Validation des entrées** présente sur inscription, commandes, statuts admin, reset password.
- **Transactions Prisma** sur inscription, reset password, livraisons et ledger.
- **Statuts livreur** : acceptation de livraison, progression et retour `AVAILABLE` après livraison.
- **Notifications internes** créées lors de validation restaurant/livreur.

### 4.2 Risques et recommandations

| Sévérité | Description | Localisation | Recommandation |
|----------|-------------|--------------|----------------|
| **Élevée** | Race condition possible à l'acceptation d'une livraison : `findFirst` puis `update` sans condition atomique | `src/app/api/driver/deliveries/[id]/accept/route.ts` | Utiliser `updateMany` conditionnel `id/status/driverId:null` ou transaction sérialisée |
| **Élevée** | Transitions de statut livraison peu contrôlées : un livreur peut passer directement à `DELIVERED` | `src/app/api/driver/deliveries/[id]/status/route.ts` | Imposer une machine d'état stricte |
| **Moyenne** | `prisma.user.update({ data: {} })` inutile pour les statuts non livrés | `src/app/api/driver/deliveries/[id]/status/route.ts:39` | Supprimer cette opération de transaction |
| **Moyenne** | Certaines listes admin restent limitées sans vraie pagination côté API/data helper | `getBillingNotifications`, settings audit, sponsorships | Uniformiser `page`, `limit`, `total` |
| **Moyenne** | `GET /api/admin/notifications` ignore les filtres déclarés par le helper | `src/app/api/admin/notifications/route.ts` | Lire `status`, `type`, `page`, `limit` depuis l'URL |
| **Moyenne** | Checkout commande ne vérifie pas `disableOrdersTemporarily` ni le montant minimum paramétré | `src/app/api/orders/route.ts` | Intégrer `PlatformSettings` dans la création de commande |
| **Faible** | Téléphone de livraison stocké dans `Order.note` | `src/app/api/orders/route.ts:102` | Ajouter champs dédiés sur `Order` ou `DeliveryAddressSnapshot` |

## 12. Actions prioritaires recommandées

1. **Corriger la race condition d'acceptation de livraison** avec update atomique.
2. **Imposer une machine d'état stricte** pour les livraisons.
3. **Refondre la suppression de compte** en anonymisation compatible comptabilité/audit.
4. **Relier payouts, wallets, ledger et settlement batches** dans Prisma.
5. **Migrer le rate limiting vers Redis/Upstash**.
6. **Créer des tables KYC** (`VerificationRequest`, `VerificationDocument`) et garder Supabase seulement pour les fichiers.
7. **Sécuriser les uploads KYC** : MIME, taille, extension, RLS vérifiée.
8. **Gérer correctement le ledger cash/card** selon livraison, paiement et statut final.
9. **Supprimer les fallbacks prod d'URL** (`APP_URL`) et valider les variables d'environnement critiques.
10. **Documenter le projet** : README setup complet, env, scripts, comptes de test, workflow de migration.

---

## 13. Conclusion

DalleUp est maintenant proche d'un MVP avancé : les principaux modules existent, le build passe, les rôles sont bien séparés et plusieurs points du précédent audit ont été corrigés. Les risques restants sont surtout liés à la robustesse production : concurrence sur les livraisons, intégrité financière, conformité de suppression des données, scalabilité du KYC et rate limiting distribué.

La prochaine étape la plus rentable est de sécuriser le **module livreur + ledger** car c'est le cœur opérationnel de la plateforme.
